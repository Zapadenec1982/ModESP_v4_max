## 1. Загальна архітектура

Бізнес-модулі **не керують реле напряму**. Кожен модуль публікує **запити** (requests)
через SharedState, а Equipment Manager виконує арбітраж і керує обладнанням.

```
┌─────────────┐   ┌─────────────┐   ┌─────────────┐
│ Thermostat  │   │   Defrost   │   │ Protection  │
│ (priority 2)│   │ (priority 2)│   │ (priority 1)│
│             │   │             │   │             │
│ req.comp    │   │ req.comp    │   │ lockout     │
│ req.evap_fan│   │ req.defrost │   │             │
│ req.cond_fan│   │ _relay      │   │             │
│             │   │ req.evap_fan│   │             │
│             │   │ req.cond_fan│   │             │
└──────┬──────┘   └──────┬──────┘   └──────┬──────┘
       │                 │                  │
       └────────┬────────┘                  │
                ▼                           │
   ┌────────────────────────┐               │
   │   Equipment Manager    │◄──────────────┘
   │     (priority 0)       │
   │                        │
   │  Арбітраж:             │
   │  1. Protection LOCKOUT │
   │  2. Defrost active     │
   │  3. Thermostat         │
   │                        │
   │  Інтерлоки:            │
   │  - defrost_relay ↔     │
   │    компресор            │
   │                        │
   │  Anti-short-cycle:     │
   │  - MIN_OFF = 180с      │
   │  - MIN_ON  = 120с      │
   └────────────────────────┘
```

**Пріоритет арбітражу:**
1. `protection.lockout = true` → все OFF (найвищий)
2. `defrost.active = true` → використовуються `defrost.req.*`
3. Інакше → використовуються `thermostat.req.*`

## 2. Модуль Thermostat — охолодження

### 2.1. State machine

```
         ┌────────────────────────────────────┐
         │              STARTUP               │
         │    (очікування startup_delay)       │
         └──────────────┬─────────────────────┘
                        │ таймер закінчився
                        ▼
         ┌──────────────────────────────────┐
    ┌────│              IDLE                │◄──────────────┐
    │    │  компресор OFF, очікування       │               │
    │    └──────────────┬───────────────────┘               │
    │                   │ T >= SP + diff                    │
    │                   │ AND пауза >= min_off_time         │
    │                   ▼                                   │
    │    ┌──────────────────────────────────┐               │
    │    │            COOLING               │               │
    │    │  компресор ON, охолодження       │───────────────┘
    │    └──────────────────────────────────┘  T <= SP
    │                                          AND робота >= min_on_time
    │    sensor1_ok = false
    │                   │
    │                   ▼
    │    ┌──────────────────────────────────┐
    │    │          SAFETY_RUN              │
    │    │  циклічна робота компресора      │
    │    │  ON: safety_run_on хв            │
    │    │  OFF: safety_run_off хв          │
    └────│  (при відновленні → IDLE)        │
         └──────────────────────────────────┘
```

### 2.2. Асиметричний диференціал

Контролер використовує **асиметричний** диференціал:

- **Увімкнення компресора:** `T_air >= setpoint + differential`
- **Вимкнення компресора:** `T_air <= setpoint`

Приклад (setpoint = -20°C, differential = 2°C):
- Компресор вмикається при T >= -18°C
- Компресор вимикається при T <= -20°C
- Робочий діапазон: від -20°C до -18°C

### 2.3. Захист компресора

Два рівні захисту працюють незалежно:

**Рівень 1 — Thermostat (програмний):**
| Параметр | Опис | Default | Діапазон | Одиниці |
|----------|------|---------|----------|---------|
| min_off_time | Мін. пауза компресора | 3 | 3..10 | хв |
| min_on_time | Мін. час роботи | 2 | 2..10 | хв |
| startup_delay | Затримка після boot | 1 | 0..5 | хв |

**Рівень 2 — Equipment Manager (апаратний):**
| Параметр | Опис | Значення |
|----------|------|----------|
| COMP_MIN_OFF_MS | Мін. пауза реле компресора | 180 с |
| COMP_MIN_ON_MS | Мін. час ON реле компресора | 120 с |

Equipment Manager захищає компресор **незалежно** від джерела запиту.
Навіть якщо defrost або інший модуль запросить вимкнення компресора раніше —
реле не перемкнеться поки не мине мінімальний час.

### 2.4. Вентилятор випарника

Три режими роботи (параметр `evap_fan_mode`):

| Режим | Опис | Логіка |
|-------|------|--------|
| 0 — Постійно | Вентилятор завжди ON | `request_evap_fan(true)` |
| 1 — З компресором | Синхронно з компресором | `request_evap_fan(compressor_on)` |
| 2 — За T випарника | Вимикається при високій T_evap | Гістерезис: OFF при `T_evap > fan_stop_temp`, ON при `T_evap <= fan_stop_temp - hyst` |

Режим 2 потребує датчик випарника (`evap_temp` binding). Якщо датчик відсутній або несправний — fallback на режим 1.

**Параметри режиму 2:**
| Параметр | Default | Діапазон |
|----------|---------|----------|
| fan_stop_temp | -25°C | -40..10°C |
| fan_stop_hyst | 2°C | 0.5..5°C |

**Під час розморозки** вентилятор випарника вимикається (thermostat paused).

### 2.5. Вентилятор конденсатора

Працює **синхронно з компресором** із затримкою вимкнення:

- Компресор ON → конденсаторний вентилятор ON (одразу)
- Компресор OFF → конденсаторний вентилятор OFF (через `cond_fan_delay` секунд)

Затримка забезпечує відведення залишкового тепла від конденсатора.

| Параметр | Default | Діапазон |
|----------|---------|----------|
| cond_fan_delay | 30 с | 0..120 с |

### 2.6. Safety Run

При відмові датчика камери (`sensor1_ok = false`) термостат переходить у режим
**циклічної роботи** компресора для збереження продуктів:

```
ON (safety_run_on хв) → OFF (safety_run_off хв) → ON → ...
```

| Параметр | Default | Діапазон |
|----------|---------|----------|
| safety_run_on | 20 хв | 1..60 хв |
| safety_run_off | 10 хв | 1..60 хв |

При відновленні датчика → повернення в IDLE.

### 2.7. Пауза під час розморозки

Коли `defrost.active = true`:

1. Всі запити термостата скидаються (компресор, вентилятори)
2. Таймери заморожені — on_update() повертається раніше
3. Equipment Manager використовує запити defrost замість thermostat

Коли розморозка завершується (`defrost.active` → false):

1. Стан примусово переходить в **IDLE**
2. Таймери `comp_off_time` і `comp_on_time` скидаються в 0
3. Нормальна логіка відновлюється: після `min_off_time` секунд паузи → COOLING

## 3. Модуль Defrost — розморозка

### 3.1. Типи розморозки

| Тип | Назва | Принцип | Обладнання |
|-----|-------|---------|------------|
| 0 | Природна | Зупинка компресора, лід тане від тепла камери | Тільки компресор |
| 1 | Електрична | Нагрівальний тен на випарнику | Реле відтайки (defrost_relay) |
| 2 | Гарячий газ | Гарячий фреон із компресора через реверсивний клапан | Компресор + реле відтайки (defrost_relay) |

### 3.2. State machine — 7 фаз

```
IDLE ──► STABILIZE ──► VALVE_OPEN ──► ACTIVE ──► EQUALIZE ──► DRIP ──► FAD ──► IDLE
         (тільки тип 2)                          (тільки тип 2)
```

Для **природної (тип 0)** та **електричної (тип 1)** відтайки:
```
IDLE ──► ACTIVE ──► DRIP ──► FAD ──► IDLE
```

### 3.3. Фази детально

#### IDLE — очікування
Перевіряє умови для запуску відтайки. Інтервальний таймер тікає.

#### STABILIZE (тільки тип 2)
Підготовка до гарячого газу. Компресор і конденсаторний вентилятор працюють,
вентилятор випарника вимкнено. Тиск стабілізується перед відкриттям клапана.

- Тривалість: `stabilize_time` (default 1 хв)
- Обладнання: компресор ON, конд. вентилятор ON

#### VALVE_OPEN (тільки тип 2)
Відкриття клапана гарячого газу при працюючому компресорі.
Короткий період для стабілізації потоку.

- Тривалість: `valve_delay` (default 3 с)
- Обладнання: компресор ON, конд. вентилятор ON, defrost_relay ON

#### ACTIVE — нагрів
Основна фаза — випарник нагрівається, лід тане.

**Завершення:**
1. **По температурі** (основна): `T_evap >= end_temp` — випарник чистий
2. **По таймеру** (безпекова): `max_duration` — запобігає перегріву

При 3 послідовних завершеннях по таймеру — інкрементується `defrost.consecutive_timeouts`
(можлива несправність нагрівача або датчика).

- Макс. тривалість: `max_duration` (default 30 хв)
- Обладнання: залежить від типу (див. таблицю нижче)

#### EQUALIZE (тільки тип 2)
Вирівнювання тиску після закриття клапана ГГ. Все обладнання вимкнено.

- Тривалість: `equalize_time` (default 2 хв)
- Обладнання: все OFF

#### DRIP — дренаж
Вода стікає з випарника. Все обладнання вимкнено.

- Тривалість: `drip_time` (default 2 хв)
- Обладнання: все OFF

#### FAD — охолодження після відтайки
Компресор і конденсаторний вентилятор ON, вентилятор випарника OFF.
Випарник охолоджується перед відновленням нормальної роботи.
Якщо ввімкнути вентилятор випарника одразу — він рознесе залишкову вологу по камері.

**Завершення:**
1. **По температурі:** `T_evap < fad_temp` — випарник охолоджений
2. **По таймеру:** `fan_delay` — максимальний час фази

- Тривалість: `fan_delay` (default 2 хв)
- Обладнання: компресор ON, конд. вентилятор ON, вент. випарника OFF

### 3.4. Ініціація розморозки

| Метод | Опис |
|-------|------|
| 0 — За таймером | Запуск через кожні `interval` годин |
| 1 — За температурою | Запуск коли `T_evap < demand_temp` |
| 2 — Комбінований | Перший з: таймер АБО demand |
| 3 — Вимкнено | Тільки ручний запуск через WebUI |

**Лічильник інтервалу:**
- `1` — реальний час (таймер тікає завжди)
- `2` — час компресора (тікає тільки коли компресор ON)

**Оптимізація:** якщо `T_evap > end_temp` на момент trigger — відтайка пропускається
(випарник чистий, немає потреби).

**Demand trigger** має мінімальний інтервал = `interval / 4`, щоб запобігти
занадто частим відтайкам відразу після попередньої.

### 3.5. Ручне керування

- **Запуск:** `defrost.manual_start = true` (через WebUI або MQTT)
- **Зупинка:** `defrost.manual_stop = true` — перериває відтайку на будь-якій фазі

При ручній зупинці відтайка завершується одразу (перехід в IDLE),
без проходження DRIP/FAD фаз.

### 3.6. Збереження стану (NVS Persist)

Два runtime параметри зберігаються в NVS і переживають перезавантаження:
- `defrost.interval_timer` — поточний стан інтервального таймера
- `defrost.defrost_count` — кількість виконаних відтайок

## 4. Взаємодія модулів

### 4.1. Порядок виконання в main loop

```
Equipment Manager (priority 0)   — читає сенсори, арбітрує, керує реле
Protection        (priority 1)   — моніторить аварії
Thermostat        (priority 2)   — регулювання температури
Defrost           (priority 2)   — цикл розморозки
```

### 4.2. Арбітраж Equipment Manager

```cpp
if (protection.lockout)     → все OFF
else if (defrost.active)    → використовуй defrost.req.*
else                        → використовуй thermostat.req.*
```

### 4.3. Інтерлоки (hardcoded, не обходяться)

1. **defrost_relay + компресор** одночасно → компресор OFF (для електричного типу)

Інтерлоки виконуються **після** anti-short-cycle і мають фінальне слово.

### 4.4. Потік даних при старті відтайки

```
1. HTTP POST /api/settings → defrost.manual_start = true
2. Defrost.on_update():
   - enter_phase(STABILIZE)           — тип 2 (гарячий газ)
   - defrost.active = true
   - defrost.req.compressor = true
   - defrost.req.cond_fan = true
3. Thermostat.on_update():
   - бачить defrost.active = true
   - скидає свої req.* → false
   - виходить з update (paused)
4. Equipment.on_update() (наступний цикл):
   - бачить defrost.active = true
   - використовує defrost.req.*
   - компресор і конд. вент. залишаються ON
```

## 7. Типові сценарії

### Сценарій 1: Нормальне охолодження

```
Boot → STARTUP (60с) → IDLE
  Температура росте: T = setpoint + diff → COOLING → компресор ON
  Температура падає: T = setpoint → IDLE → компресор OFF (через min_on_time)
  Цикл повторюється
```

### Сценарій 2: Автоматична відтайка теном (тип 1, ініціація за таймером)

```
COOLING → [interval 8 годин] → Defrost ACTIVE:
  - Thermostat paused (всі req = false)
  - Defrost relay ON, компресор OFF
  - T_evap зростає → end_temp досягнуто → DRIP (2 хв) → FAD (2 хв)
  - FAD: компресор ON, конд.вент ON, вент.вип OFF
  - T_evap падає < fad_temp → IDLE
  - Thermostat resumed → IDLE → (пауза min_off_time) → COOLING
```

### Сценарій 3: Відтайка гарячим газом (тип 2)

```
COOLING → [trigger] → Defrost:
  STABILIZE (1 хв): компресор ON, конд.вент ON, вент.вип OFF
  VALVE_OPEN (3 с): + клапан ГГ ON
  ACTIVE: компресор ON, клапан ГГ ON, конд.вент OFF
  → T_evap >= end_temp →
  EQUALIZE (2 хв): все OFF
  DRIP (2 хв): все OFF
  FAD (2 хв): компресор ON, конд.вент ON
  → IDLE → Thermostat resumed
```

### Сценарій 4: Відмова датчика камери

```
COOLING → sensor1_ok = false → SAFETY_RUN:
  - Компресор ON 20 хв → OFF 10 хв → ON 20 хв → ...
  - Датчик відновився → IDLE → нормальна робота
```

### Сценарій 5: Protection lockout

```
будь-який стан → protection.lockout = true:
  - Equipment Manager: все OFF (найвищий пріоритет)
  - Активна розморозка переривається
  - Thermostat зупинений
```

## 8. Danfoss-level Improvements (Phase 11a — РЕАЛІЗОВАНО)

### 8.1. Night Setback (нічний режим) — РЕАЛІЗОВАНО

4 режими активації (параметр `thermostat.night_mode`):
- **0** — вимкнено
- **1** — за розкладом SNTP (`night_start` / `night_end`, формат HHMM)
- **2** — через дискретний вхід (`equipment.night_input`)
- **3** — ручний (через WebUI/MQTT `thermostat.is_night_active`)

`effective_setpoint = setpoint + night_setback` (night_setback default = 3°C).

| Параметр | Default | Діапазон | Persist |
|----------|---------|----------|---------|
| night_mode | 0 | 0..3 | Yes |
| night_setback | 3.0°C | 0..10°C | Yes |
| night_start | 2200 | 0..2359 | Yes |
| night_end | 600 | 0..2359 | Yes |

### 8.2. Post-defrost Alarm Suppression — РЕАЛІЗОВАНО

High Temp alarm блокується на `post_defrost_delay` хвилин після завершення розморозки.
LAL alarm НЕ блокується — захист від переохолодження завжди активний.

| Параметр | Default | Діапазон | Persist |
|----------|---------|----------|---------|
| post_defrost_delay | 30 хв | 0..120 хв | Yes |

### 8.3. Display During Defrost — РЕАЛІЗОВАНО

Параметр `thermostat.display_defrost`:
- **0** — реальна T_air
- **1** — "заморожена" T_air (значення перед початком defrost)
- **2** — символ "-d-" (Danfoss style)

Результат публікується в `thermostat.display_temp` для Dashboard.
