# ModESP v4 — Можливості прошивки

**Модульний фреймворк для промислових ESP32-контролерів холодильного обладнання. Замінює Danfoss/Dixell дешевим ESP32 із професійною архітектурою.**

> Побудовано на ESP-IDF 5.5, C++17, ETL, FreeRTOS — працює на ESP32-WROOM-32 (4MB flash).

---

## Загальний огляд

| Показник | Значення |
|----------|----------|
| Ключі стану | 126 (63 STATE_META, 50 MQTT publish, 62 MQTT subscribe) |
| Модулі | 5 (equipment, thermostat, defrost, protection, datalogger) |
| Драйвери | 6 (ds18b20, ntc, relay, digital_input, pcf8574_relay, pcf8574_input) |
| HTTP-ендпоінти | 23 REST + OTA upload |
| Тести | 491 (181 host C++ doctest / 418 assertions + 310 pytest) |
| WebUI | 80KB gzip (64KB JS + 16KB CSS), Svelte 4 |
| Прошивка | ~1.2MB binary, 77–90KB free heap |
| Захист | 10 моніторів аварій, 2-рівнева ескалація |
| Ліцензія | PolyForm Noncommercial 1.0.0 |

## 1. Manifest-Driven архітектура

Єдине джерело істини — JSON-маніфести модулів і драйверів. Python-генератор створює 5 артефактів із маніфестів.

### Кодогенерація

```
module manifest.json ─┐
driver manifest.json ─┼→ Python generator → 5 артефактів
board.json + bindings.json ─┘
```

**Згенеровані артефакти:**
- `ui.json` — конфігурація WebUI (групи, параметри, типи, переклади)
- `state_meta.h` — STATE_META масив (63 записи з типами, одиницями, групами, MQTT-прапорцями)
- `module_registry.h` — реєстрація модулів у системі
- `mqtt_topics.h` — MQTT-топіки publish/subscribe
- `persist_keys.h` — ключі NVS для збереження параметрів

### Маніфест модуля

Кожен маніфест визначає:
- Ключі стану з типами (`float`, `bool`, `int`, `string`)
- Параметри з діапазонами, значеннями за замовчуванням, одиницями
- MQTT publish/subscribe напрямок
- Прапорці: `writable`, `persist`, `visible`
- Групування для UI та i18n-ключі

### Переваги
- Додавання нового параметра — лише зміна JSON, без редагування C++
- Неможливо розсинхронізувати UI, MQTT, NVS та C++ код
- Валідація маніфестів на етапі генерації

## 2. Керування холодильним обладнанням

Повний контроль холодильного циклу — термостат із 4-становим FSM, розморозка із 7-фазним FSM.

### Термостат

- **4-становий FSM** — startup, idle, cooling, safety_run
- **Асиметричний диференціал** — окремі пороги вкл/викл (наприклад, +2°C вкл, -1°C викл)
- **Нічний режим (night setback)** — автоматичне підвищення уставки в нічні години для економії енергії
- **Safety run** — примусове охолодження при втраті датчика (налаштовуваний інтервал вкл/викл)
- **Мінімальний час роботи/простою компресора** — захист від коротких циклів
- **Затримка запуску** — конфігурований час очікування після подачі живлення

### Розморозка

- **7-фазний FSM** — idle, pre_drip, heating, drip, post_drip, fan_delay, lockout
- **3 типи** — natural (зупинка компресора), electric (нагрівач), hot-gas (гарячий газ)
- **4 способи ініціації** — за інтервалом, за розкладом, за температурою випарника, ручний
- **Drip-фази** — конфігуровані затримки для стікання конденсату до і після нагріву
- **Fan delay** — затримка увімкнення вентилятора після розморозки (запобігає вкиду теплого повітря)
- **Термінація** — за температурою випарника або за таймером (що настане першим)

## 3. Equipment Manager

HAL-абстракція між логічними пристроями та фізичним залізом.

### Архітектура

- **Логічні пристрої** — compressor, evap_fan, cond_fan, heater, light, door_sensor, buzzer
- **Прив'язка через bindings.json** — кожен логічний пристрій → конкретний драйвер + канал
- **board.json** — опис плати: GPIO, I2C-адреси, кількість каналів

### Арбітраж relay

Модулі не керують relay напряму — Equipment Manager вирішує, хто має пріоритет:

```
Protection LOCKOUT > compressor_blocked > Defrost active > Thermostat
```

- Protection може заблокувати компресор (lockout або soft block)
- Defrost зупиняє компресор і вмикає нагрівач
- Thermostat запитує компресор, але отримає відмову, якщо вищий пріоритет активний

### Порядок update

Строгий порядок виклику `on_update()`:
1. Equipment (пріоритет 0) — зчитування датчиків
2. Protection (пріоритет 1) — перевірка аварій
3. Thermostat + Defrost (пріоритет 2) — керування обладнанням

## 4. Система захисту

10 незалежних моніторів аварій із 2-рівневою ескалацією та CompressorTracker.

### Монітори аварій

| Серйозність | Монітори |
|-------------|----------|
| **Критична** | Висока температура, Низька температура, Відмова датчика 1, Відмова датчика 2 |
| **Попередження** | Відкриті двері, Безперервна робота, Збій охолодження (pulldown) |
| **Інформаційна** | Швидкість зміни (rate rise), Короткий цикл, Швидкий цикл |

### 2-рівнева ескалація

- **Рівень 1 (Warning)** — попередження через MQTT та DataLogger, аварія активна
- **Рівень 2 (Lockout)** — блокування компресора, потребує ручного скидання
- Конфігуровані затримки для кожного рівня (запобігання хибним спрацюванням)

### CompressorTracker

- Відстеження часу роботи та простою компресора
- Підрахунок циклів (для моніторів short_cycle та rapid_cycle)
- Визначення безперервної роботи (для монітора continuous_run)
- Зберігається між перезавантаженнями через NVS

### Конфігурація

- 17 persist-параметрів у NVS (пороги, затримки, увімкнення/вимкнення кожного монітора)
- 38 ключів стану (активність аварій, таймери, лічильники)
- Кожен монітор вмикається/вимикається незалежно

## 5. DataLogger

6 каналів телеметрії, 18 типів подій, LittleFS для зберігання на flash.

### Телеметрія

- **6 каналів** — температура повітря, температура випарника, температура конденсатора, уставка, стан компресора, стан розморозки
- **Інтервал запису** — налаштовуваний (за замовчуванням 5 хвилин)
- **Кільцевий буфер** — автоматичне видалення найстаріших записів при заповненні
- **Потоковий JSON** — читання без повного завантаження файлу в RAM

### 18 типів подій

- Компресор вкл/викл, розморозка старт/кінець
- 10 аварій піднято/знято (high_temp, low_temp, sensor1, sensor2, door, continuous_run, pulldown, rate_rise, short_cycle, rapid_cycle)
- ALARM_CLEAR — зняття аварії з міткою часу та тривалістю
- Системні: boot, restart, OTA

### Зберігання

- **LittleFS** — wear-leveling файлова система для SPI flash
- Окремі файли для кожного каналу телеметрії
- Ротація файлів при досягненні максимального розміру

### SVG-графіки

- Генерація SVG-графіків температури безпосередньо на ESP32
- Відображення через WebUI без JavaScript-бібліотек для графіків
- Накладення подій (компресор, розморозка) на графік

## 6. Веб-інтерфейс

Svelte 4 SPA, 80KB gzip, WebSocket real-time оновлення, bento card дизайн.

### Технічні характеристики

- **Розмір** — 64KB JS + 16KB CSS (gzip)
- **Фреймворк** — Svelte 4 (компіляція в ванільний JS)
- **Збірка** — Vite + Rollup, деплой у SPIFFS/LittleFS
- **Підключення** — WebSocket для real-time оновлень стану

### Сторінки

| Сторінка | Опис |
|----------|------|
| **Головна** | Bento-card сітка з температурами, станом обладнання, аваріями |
| **Термостат** | Уставка, диференціал, нічний режим, FSM-стан |
| **Розморозка** | Тип, інтервал, фази, стан FSM |
| **Захист** | 10 моніторів, пороги, затримки, ескалація |
| **DataLogger** | SVG-графіки, журнал подій, вибір каналу |
| **Система** | WiFi, MQTT, OTA, інформація про пристрій, вільний heap |
| **Мережа** | WiFi STA/AP налаштування, MQTT-параметри, DNS |

### UX

- **GroupAccordion** — згортувані секції параметрів за категоріями
- **Bento cards** — плиткова сітка з ключовими показниками на головній
- **i18n (4 мови)** — українська, англійська, німецька, польська. Lazy-load language packs з LittleFS (~8KB gzip). Додати мову = файли перекладів, без зміни коду
- **WebSocket** — миттєве оновлення без опитування (дельта-публікація змінених ключів)
- **Адаптивність** — мобільний layout

## 7. Підключення

WiFi STA/AP, MQTT із TLS, HTTP REST API (23 ендпоінти), WebSocket.

### WiFi

- **STA-режим** — підключення до існуючої мережі з автоперепідключенням
- **AP-режим** — точка доступу для початкового налаштування
- **AP→STA probe** — періодична спроба підключення до STA з AP-режиму (backoff 30с→5хв, heap guard 50KB, timeout 15с)
- **STA watchdog** — перезапуск при відключенні STA >10 хвилин
- **mDNS** — `modesp-{id}.local` для виявлення в локальній мережі
- **Код країни UA** — канали 1–13

### MQTT

- **TLS** — mqtts:// порт 8883, перевірка серверного сертифіката
- **Delta-publish** — публікація лише змінених ключів (кеш попередніх значень)
- **Heartbeat** — кожні 30 секунд (версія прошивки, uptime, free heap, WiFi RSSI)
- **LWT (Last Will and Testament)** — автоматичне сповіщення хмари при розриві з'єднання
- **Tenant-aware топіки** — `modesp/v1/{tenant}/{device_id}/state/{key}`
- **QoS 1** — гарантована доставка для аварій (retain)
- **Exponential backoff** — 5с→300с при втраті з'єднання
- **HA Auto-Discovery** — автоматична реєстрація в Home Assistant

### HTTP REST API

- **23 ендпоінти** — конфігурація, стан, параметри, управління, діагностика
- **JSON API** — всі відповіді у JSON-форматі
- **OTA upload** — завантаження прошивки через POST multipart
- **Без автентифікації** — локальна мережа (опціонально basic auth)

### WebSocket

- **Real-time state** — push оновлень при зміні ключів стану
- **Дельта-формат** — лише змінені ключі, без повного стану
- **Множинні клієнти** — підтримка кількох одночасних підключень

## 8. OTA-оновлення

Dual-partition з відкатом, SHA256-верифікація, перевірка сумісності плати.

### Архітектура

- **Dual-partition** — дві OTA-партиції (ota_0 / ota_1), активна перемикається після оновлення
- **Автоматичний відкат** — якщо нова прошивка не підтверджена, повернення до попередньої при перезавантаженні
- **SHA256** — контрольна сума перевіряється перед застосуванням

### Перевірки безпеки

- **Board check** — прошивка містить ідентифікатор плати, ESP32 відхиляє несумісну версію
- **Розмір** — перевірка вільного місця в OTA-партиції перед записом
- **Версія** — перевірка що нова версія відрізняється від поточної

### Канали оновлення

- **HTTP upload** — завантаження .bin через WebUI або REST API
- **MQTT-команда** — URL для завантаження від хмари (ModESP Cloud або AWS IoT Jobs)
- **Статус** — репортинг прогресу та результату через MQTT

## 9. Хмарна інтеграція

Compile-time вибір хмарного бекенду через Kconfig. Два варіанти: ModESP Cloud (Mosquitto) та AWS IoT Core.

### ModESP Cloud (за замовчуванням)

- **Mosquitto MQTT** — стандартний MQTT-брокер із TLS
- **Провізіонінг** — bootstrap-ключ → автоматичні MQTT-облікові дані
- **Підписані прошивки** — OTA через хмару з SHA256-верифікацією
- **Tenant-aware** — топіки з префіксом орендаря

### AWS IoT Core (опціонально)

- **mTLS** — взаємна автентифікація через X.509 сертифікати
- **Device Shadow** — reported/desired стан синхронізований із хмарою
- **IoT Jobs** — OTA-оновлення через AWS IoT Jobs
- **Сертифікати в NVS** — upload через `/api/cloud` REST endpoint
- **Компонент** — `components/modesp_aws/` (AwsIotService як BaseModule)

### Перемикання

```
idf.py menuconfig → Cloud backend → Mosquitto / AWS IoT Core
```

Без змін для існуючих клієнтів — Mosquitto залишається за замовчуванням.

```
manifest.json → Python generator → ui.json + C++ headers → idf.py build
```

## 11. Підтримка обладнання

Підтримка плати KC868-A6 та периферії через 6 драйверів.

### Плата KC868-A6

- **MCU** — ESP32-WROOM-32, 4MB flash
- **Relay** — 6 каналів (через PCF8574 I2C)
- **Digital input** — 6 каналів (через PCF8574 I2C)
- **Аналогові входи** — NTC-термістори
- **1-Wire** — DS18B20 цифрові датчики температури

### Драйвери

| Драйвер | Тип | Протокол |
|---------|-----|----------|
| **ds18b20** | Температурний датчик | 1-Wire |
| **ntc** | Температурний датчик (термістор) | ADC |
| **relay** | GPIO relay | GPIO |
| **digital_input** | Дискретний вхід | GPIO |
| **pcf8574_relay** | I2C relay (PCF8574) | I2C |
| **pcf8574_input** | I2C вхід (PCF8574) | I2C |

### Конфігурація плати

- **board.json** — опис GPIO, I2C-адреси, кількість каналів для кожного драйвера
- **bindings.json** — прив'язка логічних пристроїв (compressor, evap_fan) до каналів драйверів
- Зміна плати — лише board.json + bindings.json, код модулів не змінюється

## 12. Досвід розробника

Manifest JSON → згенерований код. Мінімум boilerplate, максимум переносимості.

### Додавання нового модуля

1. Створити `modules/my_module/manifest.json` — визначити ключі, параметри, MQTT
2. Реалізувати C++ клас (наслідує BaseModule) — `on_init()`, `on_update()`, `on_message()`
3. Запустити `idf.py build` — генератор створює headers, UI, MQTT-топіки автоматично
4. WebUI показує новий модуль без змін фронтенду

### Zero Heap в Hot Path

- **Заборонено** — `std::string`, `std::vector`, `new`, `malloc` у `on_update()` / `on_message()`
- **Обов'язково** — `etl::string<N>`, `etl::vector<T,N>`, `etl::variant`, `etl::optional`
- Контроль через code review та host-тести

### Інструменти

- `idf.py build` — збірка з кодогенерацією
- `idf.py flash monitor` — прошивка + серійний монітор (COM15)
- Host-тести — g++ збірка без ESP32, запуск за секунди
- Pytest — повний набір інтеграційних тестів
